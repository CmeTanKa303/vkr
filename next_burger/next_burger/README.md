# Next Burger — веб-система онлайн-заказов

Прототип к ВКР «Проектирование и разработка высоконагруженной веб-системы
онлайн-заказов продукции общественного питания».

Клиент-серверное веб-приложение: витрина (SPA), конструктор бургера, корзина,
регистрация и вход по JWT, оформление и отслеживание заказов, **админ-панель**
для управления каталогом и заказами. Данные хранятся в базе и отдаются через
**REST API (JSON)**.

---

## Быстрый запуск (Windows)

1. Установите **Python 3.10+** с <https://www.python.org/downloads/>
   (при установке отметьте галочку **«Add Python to PATH»**).
2. Двойной клик по **`start.bat`**.
   При первом запуске он сам создаст окружение и поставит зависимости
   (нужен интернет, занимает пару минут), затем запустит сервер и откроет браузер.

Откроется:

| Что | Адрес |
|-----|-------|
| 🍔 Сайт (витрина)        | <http://localhost:8000> |
| 🛠 Админ-панель          | <http://localhost:8000/admin> |
| 📚 Документация API (Swagger) | <http://localhost:8000/docs> |

Остановить сервер — `Ctrl+C` в окне консоли.

### Учётные записи по умолчанию

| Роль | E-mail | Пароль |
|------|--------|--------|
| Администратор | `admin@nextburger.ru` | `admin123` |
| Пользователь (демо) | `user@nextburger.ru` | `user123` |

> Обычные пользователи могут регистрироваться на сайте сами (кнопка «Войти» → «Регистрация»).

## Запуск вручную (если не через `start.bat`)

```bat
cd backend
python -m venv .venv
.venv\Scripts\python -m pip install -r requirements.txt
.venv\Scripts\python -m uvicorn app.main:app --reload --port 8000
```

На Linux/macOS есть `start.sh` (`bash start.sh`).

Сбросить базу к начальному состоянию — `reset_db.bat` (удаляет `backend/nextburger.db`,
она пересоздастся при следующем запуске).

## Что умеет

**Витрина (`/`):**
- каталог из БД по категориям, поиск, фильтры (тип, размер, цена, добавки), сортировка;
- сторис, конструктор бургера (размер / тип / добавки) с пересчётом цены;
- корзина с сохранением между перезагрузками;
- регистрация и вход (JWT), меню пользователя, «Мои заказы» со статусами;
- оформление заказа с адресом доставки (заказ создаётся на сервере).

**Админ-панель (`/admin`):** вход только для роли `admin`.
- **Товары** — добавление / редактирование / удаление, в т.ч. товары-конструкторы
  (варианты размер+тип+цена, выбор доступных добавок), загрузка картинки файлом или по URL;
- **Ингредиенты** — CRUD;
- **Категории** — CRUD;
- **Заказы** — просмотр всех заказов и смена статуса
  (новый → готовится → доставляется → выполнен / отменён);
- дашборд со статистикой (товары, заказы, выручка и т.д.).

## Технологии

| Слой | Технология |
|------|-----------|
| Клиент | HTML + CSS + JavaScript (SPA, без сборщика) |
| Сервер | **Python + FastAPI** |
| ORM / БД | SQLAlchemy + **SQLite** |
| Аутентификация | **JWT** (PyJWT), пароли — **bcrypt** |
| Валидация | Pydantic |
| Документация API | OpenAPI / Swagger UI (`/docs`) |

> Это облегчённая (без Docker) версия стека из ВКР. PostgreSQL/Redis/RabbitMQ/Nginx
> в локальном прототипе заменены: SQLite вместо PostgreSQL. Благодаря SQLAlchemy
> переход на PostgreSQL — это смена одной строки подключения
> (`SQLALCHEMY_DATABASE_URL` в `backend/app/database.py`), прикладной код не меняется.

## Модель данных (9 сущностей, как в ВКР)

`CATEGORY`, `PRODUCT`, `VARIANT`, `INGREDIENT`, `PRODUCT_INGREDIENT`,
`USERS`, `ORDERS`, `ORDER_ITEM`, `ORDER_ITEM_INGREDIENT`.

Ключевой принцип: **цена фиксируется в позиции заказа** (`ORDER_ITEM.unit_price`,
`ORDER_ITEM_INGREDIENT.price`) — изменение цен в каталоге не затрагивает уже
оформленные заказы. Цены заказа **пересчитываются на сервере** из БД, данные
о цене с клиента не принимаются (защита от подмены).

## REST API (основное)

| Метод | Путь | Назначение | Доступ |
|-------|------|-----------|--------|
| GET | `/api/categories` | список категорий | все |
| GET | `/api/ingredients` | список ингредиентов | все |
| GET | `/api/products` | каталог, поиск, фильтры | все |
| GET | `/api/products/{id}` | детальная позиция | все |
| POST | `/api/auth/register` | регистрация | все |
| POST | `/api/auth/login` | вход, выдача JWT | все |
| GET | `/api/auth/me` | текущий пользователь | JWT |
| POST | `/api/orders` | оформление заказа | JWT |
| GET | `/api/orders` | мои заказы | JWT |
| GET | `/api/orders/{id}` | заказ и его статус | JWT |
| GET/POST/PUT/DELETE | `/api/admin/products...` | управление товарами | admin |
| ... | `/api/admin/ingredients`, `/api/admin/categories` | управление | admin |
| GET | `/api/admin/orders` · PATCH `/api/admin/orders/{id}` | заказы и статусы | admin |
| POST | `/api/admin/upload` | загрузка картинки | admin |

Полный интерактивный список и возможность «потыкать» запросы — на <http://localhost:8000/docs>.

## Безопасность (как в разделе ВКР)

- пароли хранятся только как **bcrypt-хеш с солью**;
- аутентификация по **JWT**, разграничение доступа `user` / `admin`;
- **валидация входных данных** Pydantic;
- **параметризованные запросы** через ORM (защита от SQL-инъекций);
- стоимость заказа **считается на сервере**, цены с клиента игнорируются;
- секрет подписи JWT **не захардкожен**: берётся из переменной окружения `SECRET_KEY`, иначе генерируется криптослучайно при первом запуске и хранится в `backend/.secret_key` (вне репозитория);
- загрузка изображений проверяется по **реальной сигнатуре файла** (не по расширению), SVG запрещён, отдача с заголовком `X-Content-Type-Options: nosniff`;
- валидация длины и формата полей (e-mail, URL изображения, длины строк заказа), запрет небезопасных схем URL (`javascript:`, `data:`).

### Намеренно упрощено для локального прототипа

Эти меры в ВКР отнесены к **промышленной эксплуатации** (слайды «Архитектура» и «Информационная безопасность») и в локальном прототипе не разворачиваются, чтобы запуск оставался «двойным кликом»:

- **HTTPS/TLS, обратный прокси и балансировка (Nginx)** — на локальной машине не нужны;
- **rate limiting / защита от перебора** — в продакшене выполняется на Nginx;
- **PostgreSQL, Redis, RabbitMQ, Docker** — заменены на SQLite (переход — смена строки подключения, см. выше);
- **подтверждение e-mail и платёжный шлюз** — вынесены в раздел «Дальнейшее развитие».

Учётные данные администратора по умолчанию (`admin123`) заданы в `backend/app/seed.py` для удобства демонстрации — в реальной системе их следует сменить.
